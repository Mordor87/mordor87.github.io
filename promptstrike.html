LLM Red Teaming — What Traditional Offensive Security Misses About AI Systems

Part 1: How LLMs Actually Work — and why the mental models you built for software don't apply here.

BH
Bharath Haveli Bhadhu Rao
MAR 12, 2026  ·  12 MIN READ

Most security professionals approaching AI systems for the first time are asking the wrong questions. Not because they lack skill — but because the mental model built for traditional software doesn't map cleanly to systems that generate behavior.

I've spent the last few years doing security reviews, threat modeling, and security validation testing on LLM systems, and the same blind spots come up every time. This series is my attempt to close them: how LLMs work, how they break, and what it actually looks like to assess one as a security engagement.

Before diving into LLM attacks, it's important to first understand how these systems actually work. This article will cover the following:

In this article
  1. Overview of LLM Architecture
  2. The LLM Lifecycle
  3. Extending LLM Capabilities

📌 This is Part 1 of the LLM Red Teaming mini-series. Each part builds on the last, starting here with foundations, moving into attacks, and eventually into what a real security assessment looks like in practice.

§

Overview of LLM Architecture

At their core, large language models are next-token predictors. Given a sequence of tokens, they output a probability distribution over what comes next. That's it. Everything else — reasoning, instruction following, refusing harmful requests — is emergent behavior arising from training at enormous scale.

"The model doesn't 'understand' your prompt the way you understand it. It pattern-matches against trillions of training examples to predict the most statistically likely continuation."

This is where most red teamers go wrong first: they bring assumptions about deterministic logic. Traditional software has code paths. An LLM has a weight space — a continuous function trained to approximate the distribution of human-generated text.

The Transformer Architecture

Modern LLMs are built on the transformer architecture. The key mechanisms to understand from a security perspective:

  • Attention heads — relate tokens across the entire context window simultaneously
  • Context window — the max token sequence processable at once (8K–200K in frontier models)
  • Softmax temperature — controls output randomness; harder to predict at high values
  • System prompt — a privileged prefix in the context window used to configure behavior
⚡ Security Implication

The system prompt isn't a security boundary. It's text in the same context window as everything else. A model trained to follow instructions will, under some conditions, follow instructions that override it. This is the root cause of prompt injection.

§

The LLM Lifecycle

Understanding the lifecycle is critical — vulnerabilities can be introduced at any stage, and different stages require different attack strategies.

  1. Pre-training Trained on massive internet-scale corpora. The model learns language, world knowledge, and implicit biases. Backdoors can be introduced here through data poisoning.
  2. Supervised Fine-Tuning (SFT) Human-generated demonstrations teach the model to follow instructions. Fine-tune datasets are high-value targets for supply chain attacks.
  3. RLHF / Alignment Reinforcement Learning from Human Feedback steers behavior toward helpful, harmless outputs. This layer is imperfect and can often be bypassed with the right prompt structure.
  4. Deployment The model is served behind an API with a system prompt, optional tools, and retrieval pipelines. Most real-world attacks happen here.

"Alignment isn't a security control. It's a behavioral tendency. And like any behavioral tendency, it has edges."

§

Extending LLM Capabilities

Bare LLMs are rarely deployed in isolation. Production systems extend them in ways that dramatically expand the attack surface:

Retrieval-Augmented Generation (RAG)

A vector database retrieves relevant document chunks and injects them into the context window at inference time. Attack surfaces: document poisoning, embedding inversion, and indirect prompt injection via retrieved content.

Tool Use / Function Calling

The model is given access to tools — web search, code execution, API calls, database queries. The model decides which tools to call and with what arguments. This creates agentic attack chains: a prompt injection in a retrieved document can trigger arbitrary tool calls.

Multi-Agent Systems

Increasingly, LLMs orchestrate other LLMs. Trust boundaries between agents are almost universally underspecified. An attacker who controls input to one agent can often influence the behavior of the entire pipeline.
